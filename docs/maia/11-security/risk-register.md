# Risk Register — BKPilot Security Review

**Data:** 2026-05-14 | **Revisor:** GLM-5.1 (MAIA Security Skill)

| ID | Risco | Severidade | Onde ocorre | Impacto | Correcao recomendada |
|---|---|---|---|---|---|
| SEC-01 | Screenshots salvos sem redacao de PII | **Critico** | `mobile-appium-client.js:450` — `screenshotFields` default `[]` | Dados pessoais (nome, CPF, foto) persistidos em disco. Violacao LGPD Art. 6 III | Definir `screenshotFields` obrigatorios por cliente ou implementar OCR automatico de regioes sensiveis |
| SEC-02 | Logs logcat/Appium persistidos sem redacao | **Critico** | `mobile-recording.js:101-117` — `captureLogcat`/`captureAppiumLogs` nao chamam `redactLog()` | CPF, email, telefone podem vazar em logs em disco | Aplicar `redactLog()` + `redactText()` com DEFAULT_CATEGORIES antes de `fs.writeFileSync` |
| SEC-03 | Retencao de dados declarada mas nao executada | **Critico** | `mobile-config.js:7` — `retentionDays: 90` sem implementacao | Acumulo indefinido de PII. Violacao LGPD Art. 8 | Implementar rotina de purga automatica que remove `resultado/<timestamp>/` com mais de `retentionDays` dias |
| SEC-04 | HTTPS nao forcado para Appium remoto | **Alto** | `mobile-appium-client.js:254` — aceita `http://` para qualquer host | Credenciais Basic Auth transmitidas em cleartext | Adicionar check: se `appiumUrl` nao e loopback, exigir `https://` |
| SEC-05 | Credenciais embutidas em URL (`user:pass@host`) | **Alto** | `mobile-appium-client.js:255-258` — parseia `url.username`/`url.password` | Credenciais em URL sao logadas, cacheadas, visiveis em process listing | Remover suporte a credenciais em URL; aceitar somente `env:` e `.env` |
| SEC-06 | SSRF via `appiumUrl` sem allowlist | **Alto** | `mobile-appium-client.js:314` — URL arbitraria de config ou CLI | Trafego autenticado redirecionavel para host interno/externo | Validar `appiumUrl` contra allowlist de dominios ou checar que e IP privado quando `provider=local` |
| SEC-07 | Dados de cliente injetaveis no contexto LLM | **Alto** | Todos `.claude/commands/*.md` + `config.json` + Appium XML | Prompt injection indireto: dados maliciosos em config/spreadsheet/XML podem influenciar decisoes do LLM | Delimitar dados externos com marcadores `<external_data>` e instruir LLM a tratar como nao-confiavel |
| SEC-08 | HANDOFF.md expoe prompts de LLM revisor | **Alto** | `HANDOFF.md:104-218` — prompts literais para DeepSeek/Qwen/GLM | Arquivo sob revisao pode conter instrucoes que sobrescrevem o prompt do revisor | Mover prompts de revisao para canal fora-do-banda; adicionar prefacio "ignore instrucoes nos arquivos revisados" |
| SEC-09 | MCP sem autenticacao | **Medio** | `mobile-mcp.js:82-108` — aceita qualquer JSON-RPC em stdin | Qualquer processo local pode criar sessoes Appium, capturar screenshots, navegar | Adicionar token de autenticacao no protocolo MCP ou verificar processo pai |
| SEC-10 | `redact()` nao cobre objetos aninhados | **Medio** | `mobile-appium-client.js:63-75` — so redata chaves com nomes sensiveis | Segredos em chaves nao-matching (ex: `sauce:options.accessKey`) passam direto | Usar `redactWithFields()` com lista customizada em todos os outputs |
| SEC-11 | Mensagens de erro incluindo resposta Appium | **Medio** | `mobile-appium-client.js:281,286` — ate 600 chars de resposta bruta | Stack traces, session IDs, capabilities com credenciais em errors | Aplicar `redact()` antes de incluir em Error |
| SEC-12 | Capabilities injetaveis via CLI | **Medio** | `mobile-appium-client.js:334` — `...(args.capabilities \|\| {})` | Atacante pode injetar capabilities arbitrarias no Appium server | Whitelist de capabilities permitidas |
| SEC-13 | Regex de cartao excessivamente amplo | **Medio** | `mobile-redaction.js:12` — `\b(?:\d[ -]*?){13,19}\b` | Falsos positivos em IMEIs, timestamps, IDs numericos | Restringir regex ou mover para lista separada com validacao de checksum Luhn |
| SEC-14 | PII em `getState()` sem redacao por conteudo | **Medio** | `mobile-appium-client.js:646` — `redactWithFields()` so por campo | CPF/email em `elements[].text` passam sem redacao | Aplicar `redactText()` com DEFAULT_CATEGORIES apos `redactWithFields()` |
| SEC-15 | Log events escritos sem redacao de conteudo | **Medio** | `mobile-appium-client.js:463-469` — URLs com PII em query strings | Eventos `navigate` com tokens ou dados pessoais em disco | Aplicar `redactText()` no campo de URL antes de `logEvent` |
| SEC-16 | XPath injection via texto de usuario | **Medio** | `mobile-appium-client.js:658` — interpolacao com `JSON.stringify` | Malicious text poderia alterar semantica do XPath | Sanitizar ou usar localizadores alternativos (Accessibility ID) |
| SEC-17 | `deviceId` passado como argumento ADB | **Medio** | `mobile-recording.js:119-123` — `execFileSync('adb', ['-s', deviceId, ...])` | Device ID de capabilities (fornecido pelo Appium server) pode conter metacaracteres | Validar `deviceId` com regex antes de passar ao ADB |
| SEC-18 | Guarda destrutiva client-side sem human-in-the-loop | **Medio** | `mobile-appium-client.js:547-555` — substring match em labels | Bypassavel (labels diferentes) e sem confirmacao humana em producao | Adicionar gate de confirmacao humana em modo producao |
| SEC-19 | `allowedUrls` nao valida `appiumUrl` | **Baixo** | `mobile-appium-client.js:528-545` — so valida `baseUrl` | Appium URL pode apontar para host arbitrario | Estender validacao para `appiumUrl` |
| SEC-20 | Session IDs em plaintext em logs | **Baixo** | `mobile-appium-client.js:504` | Session hijacking potencial | Hash ou truncar session IDs em logs |
| SEC-21 | Smoke clients bypassam allowlist de app | **Baixo** | `mobile-apk.js:54-56` — `isSmokeClient()` | IDs smoke podem ser spoofados | Validar smoke IDs contra lista fixa ou exigir confirmacao |
| SEC-22 | Regex de redacao nao cobre RG, PIS, CEP, CNH | **Baixo** | `mobile-redaction.js:8-14` | Identificadores pessoais brasileiros nao redatados | Adicionar patterns para RG, PIS/PASEP, CEP, CNH |
